# 一面

## 面试形式
前半部分采用电话面试的形式，后半部分使用在线文本编辑器进行机试。

## 面试内容
进行自我介绍后，先按照简历上所写的内容聊项目经历。本人简历上有 Flutter Web 的 MVP Demo 项目，面试官围绕这个项目的实现问了一些问题。问到了 Flutter 中的三棵树，回答对 Flutter 的运用主要停留在使用层面，没有接触过。

问完 Flutter 项目的问题后问了几个 Android 业务需求相关的问题，其中聊到了线程池：如果要自己实现一个线程池，应该如何设定最大并发数？回答根据 CPU 的线程数及任务的类型进行分配。网络请求等 IO 密集型场景并发数较大，运算密集型场景并发数较小。

本人工作场景中使用线程池较少，主要使用 Kotlin 协程，又讲了一些协程的概念。业务需求中，着重讲了 JSBridge 的需求，聊到 JSBridge 的实现方式、客户端与前端的数据互通如何实现等。飞猪的技术栈以 Hybrid 为主，比较注重这一块。

接下来是一个场景问题：假设同一个一模一样的页面，有 H5 版本和原生版本的，在哪些场景下应该使用 H5，哪些场景下使用原生？回答了一些 H5 和原生的优缺点。在面试官的理解中，H5 页面主要作为 Fallback 备用，或是用在一些需要快速迭代页面，无法等待发版的场景。

场景问题后的一道智力题：1000 瓶水，其中有 1 瓶有毒，猴子喝下有毒的水后第 2 天死亡。有 10 只猴子，问如何判断哪瓶水有毒。初步想法是二分算法，但二分无法在第 2 天就得到有毒的水。经面试官提示后想到使用二进制的解法求解。

一面的最后是一道算法题：两个 String 类型的大数相加，只考虑正整数。使用 Java 代码成功实现。

# 二面

## 面试形式
二面全部采用电话面试的形式，没有机试。

## 面试内容
二面一开始是自我介绍，然后开始聊简历上的项目。聊项目这块儿也是 focus 在了 Flutter Web 项目，主要聊了技术选型、Flutter 与其他方案（如 H5）等的优势，如何衡量这件事情的 ROI 成果等内容。

聊完 Flutter 后谈到简历上的代码重构，简单表述了重构前和重构后的区别，然后同样是 focus 在 ROI 方面。问到这件事情的成果如何量化表示，如果再有同样的场景，是否会选择重构代码。

之后是基础知识问题：一个 Activity 中包含什么，回答了 Activity、Window、DecorWindow 的概念和关系；一个 View 上屏的流程，主要是介绍 onLayout、onMeasure、onDraw 这三个步骤。面试官的评价是讲述过程中采用的技术语言不够标准，主要是靠举例描述的（onLayout 的过程我举了 ConstraintLayout 的例子）。

后面就是一些非技术的问题了，主要谈到如何做一个需求的 Owner，描述了目前自己的工作流程，包括需求评审、排期制定、人力分配等。整体感觉回答得没有一面好，有一点被难住的感觉，目前正在等待流程中。
